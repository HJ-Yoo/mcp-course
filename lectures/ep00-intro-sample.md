## 0:00–0:40 | Hook

> **[화면: Gradio 채팅 UI, 이미 서버 연결된 상태]**

**스크립트:**

여러분, 이 대화를 한번 보시죠.

> **[화면에 타이핑]**
> "노트북 재고가 몇 대 남았어?"

> **[AI가 자동으로 MCP tool 호출 → 실제 CSV에서 검색 → 결과 반환]**
> "MacBook Pro 14인치 M3 Pro 45대, Dell Latitude 5540 30대, Lenovo ThinkPad X1 Carbon 12대가 재고에 있습니다."

평범한 챗봇 같죠? 그런데 **오른쪽 패널**을 보세요.

> **[MCP 로그 패널 하이라이트]**
> ```
> ✅ Call 1: lookup_inventory (23ms)
> Args: {"query": "노트북"}
> Result: [{"name": "MacBook Pro 14\" M3 Pro", "quantity": 45, ...}, ...]
> ```

AI가 스스로 판단해서 `lookup_inventory`라는 **도구를 호출**했습니다.
대답을 지어낸 게 아니라, **실제 데이터**를 조회한 겁니다.

이게 바로 **MCP**입니다.

## 0:40–1:30 | 문제 제기 — 말만 하는 AI

**스크립트:**

ChatGPT든 Claude든, LLM은 놀랍도록 똑똑합니다.
하지만 치명적인 한계가 하나 있어요.

> **[슬라이드: 벽 앞에 서 있는 AI 캐릭터]**

**혼자서는 아무것도 못 합니다.**

- "재고가 몇 개야?" → 모릅니다. 데이터베이스에 접근할 수 없거든요.
- "티켓 만들어줘" → 못 합니다. 시스템에 쓰기 권한이 없거든요.
- "보안 정책이 뭐야?" → 지어냅니다. 회사 문서를 읽을 수 없거든요.

LLM은 **뇌는 있는데 손과 눈이 없는** 상태입니다.

그래서 각 회사마다 커스텀 플러그인을 만들어 왔습니다.
ChatGPT용 하나, Claude용 하나, Gemini용 하나...

> **[슬라이드: N×M 문제 다이어그램]**
> ```
> 5개 LLM × 20개 서비스 = 100개 커스텀 통합 😱
> ```

이건 유지보수 지옥입니다.

## 1:30–2:30 | 30초 만에 이해하는 MCP

**스크립트:**

MCP — **Model Context Protocol** — 은 이 문제의 해답입니다.

> **[슬라이드: USB-C 비유]**

스마트폰 충전기 기억나시죠?
예전에는 기기마다 다른 케이블이 필요했습니다.
Lightning, Micro USB, 전용 충전기...

**USB-C가 모든 걸 통일했습니다.**

MCP는 AI 세계의 USB-C입니다.

> **[슬라이드: 1×(N+M) 다이어그램]**
> ```
> LLM들 ──→ [MCP 표준] ←── 외부 서비스들
>
> 5 + 20 = 25개 구현이면 전부 연결! ✅
> ```

서버 하나 만들면, **OpenAI에서도, Claude에서도, Cursor에서도** 그대로 작동합니다.

MCP 서버가 제공하는 것은 딱 세 가지입니다:

| 구성요소 | 역할 | 비유 |
|---------|------|------|
| **Tools** | AI가 실행하는 함수 | 🤲 손 |
| **Resources** | AI가 읽는 데이터 | 👁️ 눈 |
| **Prompts** | 재사용 가능한 지시문 | 📋 대본 |

이 세 가지면 AI에게 **손과 눈과 대본**을 줄 수 있습니다.

## 2:30–4:30 | 라이브 데모 — 실제 동작하는 사내 AI 비서

**스크립트:**

백문이 불여일견. 직접 보여드리겠습니다.

이 과정에서 함께 수행해볼 프로젝트 — **Acme Corp Internal Ops Assistant** — 입니다.

> **[화면: Gradio UI 전환]**

### 데모 1: 재고 조회 (Tool)

> **[타이핑]** "헤드셋 재고 알려줘"

> **[결과]** AI가 `lookup_inventory` Tool을 호출해서
> Jabra Evolve2 75 (40개), Poly Voyager Focus 2 (20개) 등 실제 재고를 반환합니다.

**포인트:** AI가 "헤드셋"이라는 자연어를 이해하고, 스스로 검색 쿼리를 구성해서 도구를 호출했습니다.

### 데모 2: 정책 문서 검색 (Tool + Resource)

> **[타이핑]** "원격근무할 때 인터넷 비용 지원이 되나요?"

> **[결과]** AI가 `search_policy` Tool로 관련 정책을 찾고,
> `policy://remote-work` Resource로 전문을 읽은 후,
> "월 $75까지 인터넷 비용이 지원됩니다 (Section 5.2)" 라고 **출처를 인용**해서 답합니다.

**포인트:** 지어낸 게 아닙니다. 실제 정책 문서의 특정 섹션을 인용하고 있습니다.

### 데모 3: 티켓 생성 (Tool — 확인 게이트)

> **[타이핑]** "모니터가 깜빡거려서 수리 티켓 만들어줘"

> **[결과]** AI가 먼저 **프리뷰**를 보여줍니다:
> "Preview: 'Monitor flickering issue' (priority: medium). 생성할까요?"

> **[타이핑]** "응, 만들어줘"

> **[결과]** AI가 `confirm=True`로 다시 호출 → 실제 티켓 생성됨
> TKT-006번 티켓이 JSONL 파일에 저장됩니다.

**포인트:** 위험한 동작(데이터 변경)은 반드시 **사용자 확인을 거칩니다**. 이것이 MCP의 "확인 게이트" 패턴입니다.

> **[MCP 로그 패널 전체 보여주기]**

오른쪽 로그를 보시면, AI가 이번 대화에서 총 4번의 MCP 도구 호출을 했고,
각 호출의 입력값, 결과, 응답 시간이 모두 기록되어 있습니다.

**이 전체 시스템을 이 과정에서 처음부터 끝까지 만듭니다.**

## 강사 노트

### 사전 준비
1. Gradio UI 실행 상태로 시작 (`uv run python ui/app.py`)
2. MCP 서버 stdio 연결 완료 상태
3. ANTHROPIC_API_KEY 설정 확인
4. 데모 순서: 재고 → 정책 → 티켓 (점진적 복잡도)

### 데모 백업
만약 API 호출이 느리거나 실패할 경우:
- 스크린 녹화본 준비 (사전에 녹화)
- MCP Inspector로 직접 tool call 시연 가능

### 톤 & 페이스
- 처음 40초가 가장 중요 — **"이 대화, 뭔가 다릅니다"**로 호기심 유발
- 데모는 빠르게, 설명은 짧게 — 5분이므로 긴장감 유지
- 기술 용어 최소화 — "JSON-RPC", "Transport" 등은 EP01부터 설명
- 마지막 30초에 **과정 전체 로드맵**을 보여주고 기대감 형성
